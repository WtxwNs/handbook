# 东南大学未来机器人 · AI 科研生存手册

> 这份手册写给和我一样、打算用**科研产出**来提升**保研竞争力**，或者只是希望本科进入科研的未来机器人同学。当然泛CS都可以看看，作为借鉴。
> 显然这并不是一套你可以照着做的TODO List，因为具体学什么、做什么项目，取决于你到底想做什么方向。每一个方向都有非常成熟的教程，不必我多讲。
> 我只提供一套**路径规划与生存机制**：怎么更早进入科研、怎么避免空转、怎么把第一篇文章搞闭环、以及时间窗口如何预留。

---

## 0. 前提

1. **我们先无视课程**
   培养方案能告诉你这个专业的能力体系，但科研节奏不会等待你的课程。等你到时候上完了对应课程，你就会发现已经没时间了。这一点也适用于想打电赛、打RM/RC的同学。你不能等课程，必须自己直接去学。在未院有很多课程被直接取消、消失的情况（NLP、机器人视觉、三维视觉、具身智能这类方向课在我这届的实际供给里就很不稳定）。所以不能把“等开课”当作任何关键节点。

2. **科研只是保研的一部分**
   最终保研仍然要拼**绩点/排名**，但“已发表论文”基本属于硬通货，能显著提高底气。一篇A能带你到很好的地方。

## 1. 先看大地图：培养方案给了什么、没给什么

培养方案反正你也会有，这里讲跟科研规划相关的：

* 未来机器人这套培养体系是典型工科底盘路线：通识、专业基础、实践占比很高。
* 专业核心系统 + 机器人 + 控制 + 工程：比如自动控制原理、信号与系统、数据结构与算法、机器人智能导论、机器人力学、统一机器人学等。  

纸面上当然也列了机器学习、强化学习、具身智能、自然语言处理、机器人视觉、三维视觉等方向课，而且很多在大三/大四。
但依据我们的前提，**这些课就算开了，也只能当加成，不能当主线**。如果你要本科科研并且追求产出，主线应该是你能不能把一个科研项目推进到“可投稿”的交付物。所以第一条就是拿来告诉读者，上课对科研没用。（更：但是强化学习有用，支持大家好好写大作业）

## 2. 如何看待大一～大三

未来机器人的时间结构大概是这样：

* **大一**：不算很累，但也不闲，尤其大一下开始实验后会明显变忙。
* **大二**：非常忙，科研启动失败可能就在这里。
* **大三**：相对闲很多，但如果你大三才启动科研，往往错过产出—投稿—中稿的窗口期。

所以我的规划建议是三段式：

* **大一：强制入场**把自己变成能上手科研本科生，利用制度/导师选择把自己塞进一个能推进的科研系统
* **大二：开始工作**开始干活，在大二上学期以一作/二作的工作量完成第一份工作；大二下主导项目推进，并开始尝试提出自己的见解 or 大二下寻找RA的机会，去上海，北京，或者南大准备进外校组，当然上海可能是最好的选择。这一段你会很累，但是本科要求产出本来也不容易，坚持就是坚持。
* **大三：等待投稿周期**让结果在真正竞争前落地，拿着中稿结果在大三上/寒假开始联系老师

## 3. 最小可用能力栈

你达标的定义。

### 3.1 通用科研最低配

* **Python**：能读懂别人的 repo，能改代码，能验证输出是否符合预期。
* **Linux + Git**：能在服务器/本地稳定跑训练与评测；不会因为环境崩了、版本乱了而自爆。
* **深度学习框架**：通常是 PyTorch；能照着规范 repo 把训练、推理/生成、评测跑通。
* **论文阅读能力**：能把论文拆成“问题—方法—实验—结论—复现要点”，并快速判断它的实验套路。
这里你可以极度依赖AI，只要能达成目标，了解核心即可。

### 3.2 IF Robotics

* **ROS**：会基本开发与调试（至少能跑起来、能接传感器/控制接口）。
* **仿真平台**：至少会一个（Gazebo / MuJoCo / Isaac / 其他均可）。

> 这两个属于机器人方向的基础，不会它们你很难上手。

## 4. 怎么接触老师

### 4.1 校内正常能接触到老师的渠道

* 大二开始会有“一对一导师”的选择要求。
* 平时也会有开放互动、导师交流等活动。

但这些活动有局限：交流常常主要集中在机械/自动化/仪科学院体系里。如果想做更偏 AI 的方向并去**计算机学院**，不能等自然遇到，必须自己提前联系。

### 4.2 关键机制：把外院老师放进导师库

这点非常实用：如果提前联系到计算机学院的老师，**外院老师是可以被放进导师库**的，后续就有机会把他选为导师。
这意味着：**提前联系不是额外任务，而是会反过来决定能不能选到合适导师**。
但是实际上，是不是你的本科导师不妨碍你的工作，你无视你的导师去当其他老师的RA也可以。

### 4.3 为什么很多人卡死在计算机学院难接触

最典型的死循环是：

* 正常路径下几乎见不到计算机学院老师；
* 缺信息，不知道自己想做什么；
* 不知道找谁；
* 找到了也说不清“为什么找他”。

这不是你能力差，是信息结构的问题。你可以：

> **先拿一个科研闭环当证明**，然后再去联系更匹配的老师/组（找外校也一样）。
> 因为一旦能说清自己的能力，就不再需要费劲证明自己并祈求怜悯。

## 5. 怎么选组：青年教师更适合追产出

### 5.1 Why 青椒

青年教师（讲师/新入职助理教授/新晋副教授）往往更愿意带本科生、也更愿意让本科生走完整投稿流程。

* 研究生名额有限，本科生也能成为稳定劳动力；
* 本科生论文署名对他们是业绩；
* 他们刚经历博士阶段，离“怎么做实验—怎么写文章—怎么投稿”更近。

### 5.2 科研真正的硬门槛：有没有推进人

本科生的第一篇文章必须有一个有经验的投稿者带路，光读文章跑开源几乎不可能发现一些细节，也容易陷入迷失。
如果一个组里没有博士/高产硕士/经验投稿者来每周对齐进度，本科生想跑出论文闭环会非常难。
宁可不进组，也不要进一个“没人带节奏、没人把关写作”的组。

## 6. 第一次联系老师聊什么

不写“我热爱科研”“我很努力”这种没信息增益的话。结构可以是：

1. **投入时间**：每周能投入多少小时、可持续多久
2. **已有技术栈**：能跑通什么、做过什么（最好给链接/证据）
3. **明确诉求**：就是想产出、想走完整投稿闭环
4. **希望老师给什么**：一个可落地 idea/一个正在缺人的任务切片；我来主力推进，尽量不占老师心智

给老师的承诺只有一个：**我能稳定交付并按节奏推进**。展示你的能力和意愿。

## 7. 如何判断一个 idea 能不能做：开源要活，评测要公式化

最重要的辨认能力不是想到好 idea，而是判断一个 idea 是否能在本科窗口期做出产出。很多时候你要做的其实是可行性评估

### 7.1 baseline 不是“有开源”，而是“活的、跑得通的”

不要因为 repo 开源就相信它能用。还得看：

* star（参考，但不是决定性）
* **issue 数与质量**：有人在用、有人在报错、有人在回答/修
* **PR（pull request）**：是否有人在持续维护、是否能看出社区协作
* 最近 commit（是否长期停更）
* README 是否规范：环境/训练/推理/评测/权重/配置是否完整

**只有跑得通的 baseline 才能让你不死在工程上**。从头搭训练框架对本科生来说太容易把时间烧没。

此时你可以出门左转看看我的RepoCheck，并给它一个star。

### 7.2 评测链路要“公式化”：先跟随领域惯例

优先选择那种：

* 数据集/benchmark 是共识的
* 指标是共识的
* 大家论文里都在用这套评测链路

原因很简单：这样不用在“设计评测”上花太多精力，工作更像流水线：复现 → 对齐 → 小改 → 跑表。设计评测是一个很不稳的东西，很容易让审稿人质疑。

## 8. 最怕的不是不会，是空转

以我自己大二为例：我其实很早就准备好进组，但真正开始做有产出的工作是在大二下快结束，中间大半年几乎没有沉淀。原因主要是：

* 任务不明确/很扯淡（看一眼就知道难出成果）
* 没有人持续推进任务
* 实验阶段几乎无人协助，直到后期写文章才有人介入

所以给大家推荐的止损原则是：

> **一旦感觉开题/开工流程不对劲，就立刻多找人聊——老师、师兄、学长都行。沟通是止损工具，不是社交装饰。**

不要自我安慰再努力一点就好。如果没人回应、没人推进，就考虑换组，因为本科窗口期太贵了。

## 9. 首篇论文最小闭环

建议大家对第一篇文章的要求不是顶会必中，而是走完完整投稿闭环。

### 交付物

1. **可复现 repo（强制项）**
   环境、脚本、训练、推理/生成、评测能跑通；README 写清楚步骤与预期输出。

2. **主结果对照表（强制项）**
   同数据集、同指标、同流程，和强 baseline 比较。

3. **消融实验（强制项）**
   证明关键改动确实贡献了提升/特性。

4. **泛化或鲁棒性实验**
   换子数据集/换设置/换分布，哪怕简单也要有，避免“只在一个角落调参”。

5. **可视化/案例分析**
   让人看懂改进点：成功/失败样例分析、误差分布、机制解释等。

6. **实验记录**
   失败尝试也记录：配置、种子、版本、结果。现在其实挺看重failure case。

## 10. 我第一篇文章的例子：一个 idea 怎么从出现到落成论文（供参考）

我第一篇 idea 的起点如下：

* 时间：大概从五六月份开始，真正动手在六月份。
* 师兄给的想法：歌曲生成可以用“乐谱”作为中间态——**提示词 → 乐谱 → 歌曲**。
* 我做的第一件事：调研。我看了几天论文，然后反馈：当时我没看到有人把这条链路作为明确流程来做，或者至少没有同构方案。

然后我开始做工程落地：

* 我找了一个当时非常强的开源工程，但它是端到端（提示词→歌曲），中间解码器、训练流程、解码方法都不兼容。
* 所以我很多代码是自己从头写的：训练流程、生成流程、把想法落实成可跑系统。
* 我跑了多个模型，做对比实验。
* 评估上我遵循领域惯例：大家怎么做我先怎么做。如果提出了新的贡献维度，再额外设计评估它的实验。
* 最后写论文：这个“乐谱中间态”非常好讲故事，因为它符合直觉：用结构化中间表示来生成歌曲，听起来就合理，也方便组织叙事。

第一篇工作最有价值的点就是形成这样一个闭环：

> **方向假设（别人给也行） → 快速调研确认差异/空间 → 找 baseline → 工程跑通 → 对照/消融/泛化/案例 → 写作讲故事 → 投稿**

## 11. 第一篇工作的节奏：三到六个月一篇

我认为“做得快、节奏清晰”非常重要。我的经验是：

* **三个月做完一篇**的前提：

  * 方向工作流成熟（读论文能看出大家要证明什么、需要做哪些实验）
  * 有师兄/经验投稿者带节奏（尤其给idea，实验设计参考，写作与“讲故事”把关）
  * 能稳定跑实验并记录

* 一般最多容忍 **六个月还没出稿**：

  * 如果六个月还跑不出像样结果，要么我推进出了问题（实验没跑/没对齐/没推进），要么 idea 本身错了。
  * 这两种都不该继续耗下去。

另外一个更现实的时间账本：

* 一次会议投稿周期走完大概需要 ~3 个月（审稿/出结果）。
* 新手很难一投必中，可能经历多轮投稿/重投。
* 所以必须把投稿周期留在时间规划里，不能把所有时间都花在写完初稿上。

> **第一篇的目标不是命中率，是闭环。先跑通一次完整投稿流程，再谈冲更高目标。**

## 12. 自动化学院的资源怎么用

我接触到的自动化实验室/大组方向非常全：多智能体、强化学习、分布智能、机器人相关各种方向基本都有。
对大家而言，这意味着不缺方向，缺的是：

* 任务是否明确
* 是否有推进人
* baseline 是否活
* 评测链路是否公式化
* 我能否把它推进到论文形态

对于第一篇，**方向热不热不重要，能不能推进到闭环更重要。**
这里面比较值得注意的一点是东南大学没几个老师有带本科生发好会议（弱B~强C）/期刊（二区/C以上）一作的经验，大家这个问题怎么解决可以自己斟酌，注意你要找的最核心的一个人是能帮你反复迭代的有经验的投稿者。

## 13. 保研权重

我自己的排序（偏科研保研路线）是：

1. **绩点/排名**：底线与门票
2. **已发表论文**：硬通货
3. 其他科研证据（在投论文/开源/项目/竞赛）：看匹配与认可度，波动很大

所以我不建议走极端：

* 只卷绩点不科研，会缺上限；
* 只科研不管排名，会没门票。

守住排名底线，同时用科研产出拔高上限。

## 14. 行动脚本

### 14.1 如果现在还没进组

* 先选一个我愿意读 50 篇论文的方向（热不热不作为主要标准）。
* 找一个活 baseline（issue/PR/维护）并复现跑通。
* 写一个复现报告 + repo（这是联系老师的“证据”）。
* 联系老师，明确投入时间 + 明确想产出 + 要一个可落地任务切片。

### 14.2 如果进组了但推不动

* 立刻多找人聊：老师/师兄/学长都可以。
* 要到两样东西：

  1. 下一步可验收任务（两周内交付物是什么）
  2. 下次对齐时间（每周一次）
* 如果持续展示积极性仍无人回应，就进入换组评估，不把学期押在无人推进项目上。

## 15. AI 做科研

现在科研门槛变低了，工具很强。但我强调的会用AI是：

* 用 AI 读论文、拆实验套路、补检索关键词
* 用 AI 理解 repo、定位报错、写脚本、补文档
* 用 AI 写实验记录、整理对比表、生成写作骨架
* 最终仍然要自己保证：实验真实、复现可查、对比公平、署名规范

工具让你更快，但工具不替你负责。
